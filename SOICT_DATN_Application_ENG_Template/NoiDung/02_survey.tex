\section{Status Survey}
% GUIDE: analyse 3 sources — (i) users/customers, (ii) existing systems,
% (iii) similar applications. Compare Odoo, SAP Business One, KiotViet, Sapo, MISA
% in a table (features vs. limitations for a small distributor: cost, FEFO support,
% delivery routing, customisation). Conclude what features are required.
The requirement survey was conducted from three sources. The first source is the
typical workflow of a small or medium-sized distributor: purchasing goods from
suppliers, receiving them into warehouses, selling them to customers, issuing
invoices, and arranging delivery. The second source is the current manual
practice, where employees often use spreadsheets, paper forms, and direct
messages to coordinate stock and approvals. The third source is a group of
similar software products, including Odoo, SAP Business One, KiotViet, Sapo, and
MISA.

Existing systems confirm that lot and expiry-date management is an important
industry requirement. Odoo supports expiration-date tracking and FEFO removal
strategy \cite{odooExpiration,odooFefo}. SAP Business One batch master data
contains fields such as expiration date, manufacturing date, and admission date
\cite{sapBatch}. KiotViet, Sapo, and MISA provide Vietnamese-market functions
for managing goods by lot and expiry date
\cite{kiotvietLotExpiry,sapoLotExpiry,misaLotExpiry}. However, these products
either target broad ERP usage or retail/POS usage. A small distributor may need a
more focused workflow that connects purchase approval, goods receipt, stock
reservation, FEFO goods issue, delivery execution, invoice generation, and
failed-delivery recovery in one prototype. Table~\ref{tab:survey} compares the
surveyed products with the system developed in this thesis according to the
criteria that matter most for a small distributor.

From this survey, the required features are defined as follows. The system must
support end-to-end procure-to-stock and order-to-cash workflows, not only
standalone inventory counting. It must maintain both aggregate stock and
lot-level stock for FEFO. It must prevent overselling by reserving stock after
sales order approval. It must separate duties through role-based access control.
Finally, it must support delivery status changes and recovery when delivery
fails.

\section{Functional Overview}

\subsection{General Use Case Diagram}
% GUIDE: describe the 9 actors and the top-level use cases, then reference the figure.
\thesisfig{usecase_general}{General use case diagram of the system}{fig:uc-general}
Figure~\ref{fig:uc-general} shows the general use case diagram. The system has
nine actors. The administrator manages users, roles, master data, and system
configuration. Purchase staff create and update purchase orders, while purchase
managers approve or reject them. Sales staff create and update sales orders,
while sales managers approve or reject them. Warehouse staff confirm physical
goods receipt and goods issue, manage inventory, and monitor lots. Delivery
administrators create delivery plans, assign shippers, and monitor delivery
execution. Shippers view assigned trips and update delivery status. Accountants
view financial information, approve business documents when assigned that duty,
and manage invoice-related operations.

The top-level use cases are grouped by business area: purchasing, sales,
warehouse, delivery, reporting, and administration. This organization matches
the operational departments of a distributor and keeps each actor's
responsibility explicit.

\subsection{Detailed Use Case Diagrams}
% GUIDE: one subsection per high-level use case; the decomposed UC name must match.
\thesisfig{usecase_purchasing}{Detailed use cases — Purchasing \& Goods Receipt}{fig:uc-purchasing}
\thesisfig{usecase_sales}{Detailed use cases — Sales \& Goods Issue}{fig:uc-sales}
\thesisfig{usecase_warehouse}{Detailed use cases — Inventory \& Lot management}{fig:uc-warehouse}

The purchasing use cases in Figure~\ref{fig:uc-purchasing} include creating a
purchase order, updating it while it is still open, approving or rejecting it,
creating a goods receipt from an approved purchase order, and confirming the
receipt. The important rule is that warehouse receipt is not independent from
purchase approval; goods receipt can only proceed when the purchase order status
allows receiving.

The sales use cases in Figure~\ref{fig:uc-sales} include creating a sales
order, approving or rejecting it, reserving inventory, creating a goods issue,
confirming the goods issue, and generating the sales invoice. The sales flow is
designed to separate commercial approval from physical shipment. The warehouse
use cases in Figure~\ref{fig:uc-warehouse} include viewing inventory, managing
lots, checking expiring or expired lots, and viewing inventory transactions. The
delivery use cases in Figure~\ref{fig:uc-delivery} include creating delivery
plans, assigning shippers, viewing assigned trips, updating delivery status, and
handling failed delivery.

\subsection{Business Process}
% GUIDE: model the combined activity flow of multiple use cases (NOT a single UC).
\thesisfig{activity_procure_to_stock}{Procure-to-stock business process}{fig:act-p2s}
\thesisfig{activity_order_to_cash}{Order-to-cash business process}{fig:act-o2c}
Figure~\ref{fig:act-p2s} describes the procure-to-stock process. The process
starts when purchase staff create a purchase order for a supplier. A purchase
manager or accountant reviews the order. If the order is rejected, the process
ends with a cancellation status. If the order is approved, the warehouse can
create a goods receipt when goods arrive. On confirmation, the system validates
the received quantity, updates the purchase order receiving progress, increases
inventory, creates inventory transactions, and creates inventory lots with batch
and expiry information.

Figure~\ref{fig:act-o2c} describes the order-to-cash process. Sales staff create
a sales order for a customer and selected warehouse. A sales manager or
accountant approves the order after the system checks available stock. Approval
reserves stock so that other orders cannot consume the same quantity. Warehouse
staff then create and confirm a goods issue. Confirmation deducts physical stock
by FEFO, releases reservation, updates delivered quantity, and creates an
invoice. Delivery staff execute the delivery plan and update delivery status. If
delivery fails, the system restores stock and adjusts the related order and
invoice state.

This section specifies the use cases that carry the most important business
rules. For each use case, the actor, the pre-conditions, the main and alternate
event flows, and the post-conditions are presented in a structured form.
Tables~\ref{tab:uc-gi}, \ref{tab:uc-so}, \ref{tab:uc-gr},

order-to-cash workflows. Additional, less central use cases are specified in

\subsection{Confirm Goods Issue (FEFO)}
The goods issue confirmation use case is the outbound operation that deducts
stock and triggers invoice generation. Its specification is given in
Table~\ref{tab:uc-gi}.
\begin{table}[H]\centering
\caption{Use case specification --- Confirm Goods Issue (FEFO)}
\label{tab:uc-gi}
\begin{tabular}{|p{3cm}|p{11cm}|}
\hline
\textbf{Actor} & Warehouse staff. \\ \hline
\textbf{Pre-conditions} & Sales order is APPROVED; on-hand stock $\geq$ issued quantity. \\ \hline
\textbf{Main flow} & (i)~validate stock; (ii)~allocate the requested quantity from FEFO lots; (iii)~decrease aggregate inventory and release the reservation; (iv)~update the delivered quantity; (v)~auto-generate the sales invoice; (vi)~update the sales order status. \\ \hline
\textbf{Alternate flow} & Insufficient lot stock: the confirmation is rejected with an error and no stock is changed. \\ \hline
\textbf{Post-conditions} & Aggregate inventory and lots are reduced; a sales invoice is issued. \\ \hline
\end{tabular}
\end{table}

\subsection{Approve Sales Order (Inventory Reservation)}
The sales order approval use case is the commercial control that reserves stock
before shipment. Its specification is given in Table~\ref{tab:uc-so}.
\begin{table}[H]\centering
\caption{Use case specification --- Approve Sales Order (Inventory Reservation)}
\label{tab:uc-so}
\begin{tabular}{|p{3cm}|p{11cm}|}
\hline
\textbf{Actor} & Sales manager or accountant. \\ \hline
\textbf{Pre-conditions} & Sales order is ORDER\_OPEN; the customer is active; a warehouse is selected; order items are valid. \\ \hline
\textbf{Main flow} & (i)~load the sales order with its items; (ii)~verify that the status allows approval; (iii)~check the available quantity for each product and warehouse; (iv)~reserve inventory; (v)~set the status to ORDER\_APPROVED; (vi)~store the approver and the approval time. \\ \hline
\textbf{Alternate flow} & Insufficient available stock: the approval is rejected and the order is kept unchanged. Invalid status: the transition is rejected. \\ \hline
\textbf{Post-conditions} & The order is approved; the reserved quantity increases; the available quantity decreases; the on-hand quantity is unchanged. \\ \hline
\end{tabular}
\end{table}

\subsection{Confirm Goods Receipt}
The goods receipt confirmation use case is the inbound operation that increases
stock and creates inventory lots. Its specification is given in
Table~\ref{tab:uc-gr}.
\begin{table}[H]\centering
\caption{Use case specification --- Confirm Goods Receipt}
\label{tab:uc-gr}
\begin{tabular}{|p{3cm}|p{11cm}|}
\hline
\textbf{Actor} & Warehouse staff. \\ \hline
\textbf{Pre-conditions} & Purchase order is APPROVED or PARTIALLY\_RECEIVED; the goods receipt is DRAFT; a warehouse and receipt items are provided. \\ \hline
\textbf{Main flow} & (i)~load the goods receipt with its items; (ii)~validate the status; (iii)~check the remaining ordered quantity; (iv)~update the purchase order item received quantity; (v)~add the accepted quantity to aggregate inventory; (vi)~create an inventory transaction; (vii)~create an inventory lot with batch number and expiry date; (viii)~update the purchase order receiving status. \\ \hline
\textbf{Alternate flow} & Received quantity exceeds the remaining quantity: the confirmation is rejected. Missing batch number for a lot-managed receipt: the confirmation is rejected. \\ \hline
\textbf{Post-conditions} & The goods receipt is CONFIRMED; inventory and lots increase; the purchase order receiving progress is updated. \\ \hline
\end{tabular}
\end{table}

\subsection{Assign and Execute Delivery Trip}
The delivery trip use case covers planning by the delivery administrator and
execution by the shipper. Its specification is given in Table~\ref{tab:uc-trip}.
\begin{table}[H]\centering
\caption{Use case specification --- Assign and Execute Delivery Trip}
\label{tab:uc-trip}
\begin{tabular}{|p{3cm}|p{11cm}|}
\hline
\textbf{Actor} & Delivery administrator and shipper. \\ \hline
\textbf{Pre-conditions} & A goods issue or delivery order is ready for delivery; a shipper user exists and has the SHIPPER role. \\ \hline
\textbf{Main flow} & (i)~the delivery administrator creates a delivery plan; (ii)~selects the orders to deliver; (iii)~assigns a shipper; (iv)~the system creates the trip route and route items; (v)~the shipper views the assigned trips; (vi)~the shipper updates each delivery item as delivered or failed. \\ \hline

\end{tabular}

\section{Non-Functional Requirements}
% GUIDE: performance, reliability, usability, maintainability, security, plus
% technical requirements (PostgreSQL, REST API, JWT auth, browser support).
The system must satisfy several non-functional requirements. For performance,
common list pages such as products, customers, purchase orders, sales orders,
and inventory should respond within a few seconds on a local development
database. Critical stock updates must run in database transactions to avoid
partial updates. For reliability, approval, goods receipt confirmation, goods
issue confirmation, and failed-delivery recovery must either complete all
related changes or fail without changing business state.

For security, every protected API must require authentication, and authorization
must follow the role matrix described in Chapter~\ref{chapter:Contribution}.
Passwords must be stored using a one-way hash, and JWT secrets must be
configured outside source code in production. For usability, the web interface
must clearly show document status, validation errors, and operation results.
For maintainability, the backend must preserve the controller-service-repository
separation, DTOs must be used at the API boundary, and database schema changes
must be represented as migration scripts.

The technical requirements are as follows. The backend provides REST APIs over
HTTP and is implemented with Java and Spring Boot. The frontend is a browser
application implemented with React and Ant Design. PostgreSQL is used as the
database. The application should run in modern browsers such as Chrome, Edge,
and Firefox. The default development ports are 8080 for the backend, 5173 for
the frontend, and 5432 for PostgreSQL.
